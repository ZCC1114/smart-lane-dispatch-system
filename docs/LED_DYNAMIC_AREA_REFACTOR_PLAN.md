# LED 大屏动态区写入改造计划

本文档用于跟踪 BX6E LED 大屏写入链路改造进度。每完成一个功能点，就勾选对应条目。

## 改造目标

- 业务大屏从频繁写完整节目，改为优先使用六代卡动态区更新。
- 最新入场车辆引导信息可以更快刷新，10 秒高亮期间新车牌可以立即覆盖。
- 第 4 行提示语常驻显示：`请按照车道指示进行停车等待！`
- 保留原有完整节目写入能力，动态区失败时可以回退。
- 尽量复用控制卡 TCP 连接，减少每次连接、写入、断开的耗时。

## 当前结论

- BX6E 支持动态区。6E/6Q 系列支持 0-31 共 32 个动态区。
- 动态区内容存 RAM，适合频繁更新；普通节目写入更重，不适合高频刷新。
- 改造前业务大屏链路是 `LedGuideDisplayService -> LedScreenService.sendText(...) -> writeProgram(...)`。
- 改造前六代写入每次都会创建 `Bx6GScreenClient`、连接、写完整节目、断开。
- 本次改造优先覆盖业务引导大屏，不改变普通 LED 测试页的完整节目写入能力。

## 目标显示逻辑

### 列表模式

总入口没有新的高亮车辆时，屏幕按 1 列 4 行显示：

```text
苏B11111 驶入 1车道
苏B22222 驶入 11车道
苏B33333 驶入 3车道
请按照车道指示进行停车等待！
```

### 高亮模式

总入口拍到车牌并成功分配车道后，前 3 行居中展示一个车牌引导信息，持续 10 秒：

```text
苏B11111
驶入
11车道
请按照车道指示进行停车等待！
```

10 秒内如果又有新的入场车辆，则立即切换为新的车牌引导信息，并重新计时 10 秒。10 秒结束后恢复列表模式。

## 技术路线

### 首选方案

使用 BX6E 六代动态区更新。实际落地采用 4 个行级动态区，便于每一行独立控制字号和内容：

- 动态区 0：第 1 行。
- 动态区 1：第 2 行。
- 动态区 2：第 3 行。
- 动态区 3：第 4 行固定提示语。
- 每行只在内容、字号、颜色或区域参数变化时更新。
- 由于动态区内容存 RAM，后端会按配置周期执行一次强制全量刷新，避免控制卡重启后内容丢失。

### 连接策略

- 六代动态区写入使用可复用的 `Bx6GScreenClient`。
- 写入前检查连接状态；未连接则连接。
- 写入失败时断开并重连一次。
- 保留定时 `ping` 或写入前轻量检查，避免现场长时间空闲后连接失效。
- 应用退出时主动断开连接。

### 回退策略

- 默认优先动态区写入。
- 动态区写入失败时，允许回退到当前 `writeProgram` 完整节目写入。
- 日志需要明确区分：动态区成功、动态区失败、回退成功、回退失败。

## 风险点

- Java SDK 高层 `writeDynamic(...)` 一次主要写一个动态区；当前按 4 行最多连续写 4 次，如果现场仍然慢，需要评估底层批量动态区命令或 C SDK 批量接口。
- 动态区字体能力可能不如完整节目灵活；当前已拆成行级动态区，避免同一区域内多行字号差异受限。
- 控制卡现场固件版本可能影响动态区个别参数，需要保留完整节目写入回退。
- 长连接空闲后可能被设备或网络断开，需要重连和日志观测。
